# BisBench Run Protocol — v1

Every published run in the v1 cohort follows this protocol. The protocol is frozen before the first comparison is published. "Same prompt" is insufficient without the same starting files, tools, time budget, retry policy, viewport, and packaging requirements.

## 1. Exact prompt

All models receive the exact text in `content/prompts/<benchmark>.md`. No per-model paraphrasing, no hidden additions, no clarifications supplied to one model and withheld from another.

## 2. Starting files

Each run begins from an empty working directory. No scaffolding, no starter code, no preloaded configuration. The model produces every file.

## 3. Allowed tools

The model may use its own reasoning, write files, and (where its environment provides) read files it has written. No internet access, no retrieval over external sources, no web search, no package registry queries, no access to the BisBench repository.

## 4. Generation time limit

A single run is bounded by a wall-clock time budget of 60 minutes. A run that exceeds the budget is recorded with the status it had at the cutoff.

## 5. Retry and intervention policy

- Up to three attempts per model. Each attempt is a fresh session from the empty starting state.
- The owner does not edit, patch, or continue model output between attempts or after an attempt completes.
- The owner may answer mechanical, predeclared clarifications (e.g. "use the prompt as written") but does not provide design guidance, bug fixes, or implementation hints.
- All attempts are recorded. The selection rule (section 9) determines which attempt is published.

## 6. Dependency policy

The output must run from a static archive with no package installation and no build step. Any library the model uses must be vendored into the archive. No runtime `import` from a CDN or registry. No `npm install`, no bundler step, no transpilation runs inside BisBench.

## 7. Viewport

The output is captured and validated at a 1280×720 desktop viewport. Mobile compatibility is noted but not required for a run to be marked playable.

## 8. Packaging requirements

- `index.html` at the archive root.
- All scripts, textures, fonts, and audio included.
- No external network dependency at runtime.
- No absolute filesystem paths, no symlinks, no `../` traversal entries.
- Archive limit 25 MB; extracted limit 100 MB.
- Only browser-readable static files accepted.

## 9. Selection rule

Every attempted run is published, or a mechanical predeclared rule selects a subset. For v1, the published run is the model's first attempt that produces a browser-ready archive; if no attempt produces one, the best-ranked attempt is published with status `failed` or `partial` and a transparent run note. The owner never silently cherry-picks the best output.

## What BisBench does not measure

- Overall model quality. A single task does not rank models.
- Cost, latency, or token efficiency (deferred).
- Robustness across many tasks (v1 has one benchmark).
- Automated gameplay quality. Visitors judge the output themselves.

## Versioning

This protocol is `v1`. Each run records its `protocolVersion`. A future `v2` will be a new cohort, not a retroactive edit of v1 runs.
